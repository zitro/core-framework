**Status:** Draft — for review
**Owner:** zito

**Scope:** Product direction from v1.3.0 through v2.0.0

This is the consolidated source of truth for the next eight releases. It supersedes any

---

## 1. Vision

CORE is **the world's best Product Owner / Technical Program Manager, instantiated
per project**. It ingests everything we have access to, continuously synthesizes the
*why / what / so-what / how*, generates discovery artifacts as ranked candidates,
challenges itself for evidence, and answers questions only from the project's own
data — with citations.

It is opinionated: it bakes in **HVE + Design Thinking** as the discipline that keeps
synthesis honest, and it ships shareable artifacts as **Office documents**, not raw
markdown, so non-engineering stakeholders can actually consume them.

For the FDE triage use case, the success bar is concrete: **take a fresh FDE
assignment to a defensible Decision Brief in under 24 hours**, with a human curating
rather than authoring.

## 2. Mental model

```
Project (e.g. Discovery Pilot)
│
├─ Sources (read-mostly inputs, ingested into Documents)
│     ├─ local folder(s)
│     ├─ typed engagement repo (typed parser)
│     ├─ Microsoft Graph: Outlook / Teams / SharePoint / Calendar / People
│     └─ generic enterprise (GitHub, ADO, Jira/Confluence, web, SQL — v1.9+)
│
├─ Documents (the unified corpus, Cosmos-stored, dedup'd, hashed)
│
├─ Synthesis (auto-maintained, versioned, cited; grouped by DT mode)
│     ├─ Empathize  (Voice of Customer, Empathy Maps, Personas, Pain Points,
│     │              Stakeholder Map, Current-State Journey)
│     ├─ Define     (Problem Statements, POV, HMW, JTBD, Opportunities,
│     │              Future-State Journey, Business Value, KPIs)
│     ├─ Ideate     (Workstreams, Solution Concepts, Tech Options,
│     │              Engagement Plans, Scope Cuts, Crazy 8s capture)
│     ├─ Prototype  (Reference Architectures, Demo Briefs, Workshop Briefs,
│     │              Solution Pack draft, One-Page Brief)
│     └─ Test       (Hypothesis Tests / MVEs, Question Generator outputs,
│                    6Qs, Demo Outcome Tracker, Risks, Pre-mortems,
│                    Critic Gaps / Contradictions reports)
│
├─ The Story (composed from accepted Synthesis items;
│              exec / technical / customer-facing tone variants;
│              exported as Word / PowerPoint)
│
├─ Workshops (briefs, output capture, dot-vote synthesis)
│
└─ Chat (project-scoped RAG over Documents + accepted Synthesis,
         with citations and out-of-corpus refusal)
```

The existing four CORE phases (**Capture / Orient / Refine / Execute**) and the
existing 14-method **Methodology** page remain. They are lenses on the project
state; they are not separate workflows. The Methodology catalog is the seed for
the Generator registry — every method becomes (or backs) a Generator.

## 3. Hard rules (non-negotiable)

These rules apply across every release. PR review checks against them.

1. **Citations or it's draft.** Any Synthesis sentence without `source_refs`
   renders as draft / unsupported (muted color, "needs evidence" badge). The
   absence of evidence is itself a signal — never hidden.
2. **Generators propose, humans curate.** Every Generator output enters the
   system as `proposed`. Status moves to `accepted` or `archived` only by an
   explicit human action (with reason on archive). No silent promotion.
3. **No write-back without approval.** Vertex git commits (and any future
   write-capable connector) require an explicit human "Approve & commit" with a
   diff preview. Default behavior is read-only.
4. **Project-scoped, period.** Every read, write, retrieval, and chat answer is
   scoped by `project_id`. No cross-project leakage. Cosmos partition + RAG
   filter both enforce.
5. **Consent + provenance always.** Microsoft Graph and any future external
   source require an explicit consent dialog listing the exact mailboxes /
   chats / sites / calendars about to be read. Every Document carries source +
   uri + author + occurred_at.
6. **Confidentiality propagates.** Documents tagged `confidential` carry the
   tag into citations and into any drafted output. Critic warns when confidential
   content lands in customer-facing artifacts.
7. **Office for shareable artifacts.** Every shareable Story / Brief / Plan /
   Intake / Pack exports as `.docx` or `.pptx` using a CORE-branded (or
   customer-branded) template. Markdown is the working format inside CORE,
   Office is the deliverable format outside.
8. **Diverge before converge.** Generators always produce ≥3 candidates by
   default, on a deliberate range axis (conservative / standard / wildcard /
   inverted). Critic blocks promoting a single-candidate set into the Story
   without an explicit waiver.
9. **Falsification or waiver.** Every accepted artifact carries
   `success_criteria` (what would prove this right) and `falsification` (what
   would prove this wrong). Critic blocks acceptance without one or an explicit
   waiver-with-reason.
10. **Polish & Packaging Standard.** Every screen feels like a finished
    product: empty states with next-action, animated loading, clickable
    citation badges, real tables, semantic status colors (proposed / accepted /
    archived / draft-unsupported). No raw markdown viewers anywhere.

## 4. Sources model

Azure-only for the platform itself. Source connectors are about content
ingestion, not cloud lock-in.

### 4.1 Source plugin contract

```python
class Source(Protocol):
    kind: str                            # "local" | "vertex" | "graph_outlook" | ...
    label: str
    config_schema: type[BaseModel]
    requires_consent: bool

    async def list(self, cfg, since: datetime | None) -> AsyncIterator[Document]:
        ...
    async def fetch(self, cfg, external_id: str) -> Document | None:
        ...
    async def health(self, cfg) -> SourceHealth:
        ...
```

Each project has many `SourceBinding`s (`source_kind` + `config` blob + secret
refs in Azure Key Vault). A scheduled job (or manual "Sync") iterates bindings
and upserts Documents. Hash-based dedupe makes re-ingest cheap.

### 4.2 Document model

```python
Document {
    id, project_id,
    source_id, source_kind,        # which connector
    external_uri, external_id,     # link back to origin
    kind,                          # transcript | email | doc | issue | message | …
    authors, participants, occurred_at, ingested_at,
    title, body_md, body_text,
    facets,                        # extracted: topics, people, decisions, risks
    embedding_id, hash,
    visibility,                    # internal | confidential | public
    consent: { acquired_at, scope }
}
```

Generators / Critic / Chat work on `Document` only. Source connectors are
swappable.

### 4.4 Microsoft Graph in v1.3 — concrete requirements

This is a 2-week chunk inside v1.3 and is critical to the FDE workflow.

- **Entra app registration kit:** documented setup; required delegated scopes
  `Mail.Read`, `Chat.Read`, `Files.Read.All`, `Sites.Read.All`,
  `Calendars.Read`, `People.Read`, `User.Read`, `offline_access`. One-time
  `init` script automates app registration via `az` CLI for first-time setup.
- **OAuth2:** auth-code flow (preferred) or device-code (fallback). Refresh
  tokens stored in **Azure Key Vault**, per project, never on disk.
- **Multi-tenant:** detect customer tenant vs home tenant; show clearly in UI.
- **Consent dialog:** lists the exact resources about to be read before first
  sync; per-chat opt-in for Teams.
- **"Forget this source":** deletes Documents + revokes token; entry stays in
  audit log.

## 5. Synthesis model

### 5.1 Item lifecycle

```
proposed → accepted   (in The Story; appears as solid in UI)
         → archived   (rejected, kept for audit, with reason)
         → reviewed   (intermediate state for high-stakes items)
```

### 5.2 Common candidate envelope

```python
SynthesisItem {
    id, project_id,
    category,                      # empathize | define | ideate | prototype | test | story | ops | close | critique
    type,                          # "problem-statement", "persona", "engagement-plan", ...
    payload: dict,                 # type-specific schema
    evidence_refs: list[ref],      # to Documents, with span/quote
    generated_by: { agent, model, prompt_version, run_id },
    generation_axis: enum,         # conservative | standard | wildcard | inverted
    model_confidence: float,       # 0..1, LLM self-rated
    evidence_density: float,       # computed from refs (count, recency, source diversity)
    assumptions: list[str],        # what would have to be true
    success_criteria: list[str],   # what would prove this right
    falsification: list[str],      # what would prove this wrong
    fidelity: enum,                # paper | concept | MVE | MVP (Prototype-only)
    visibility,                    # internal | confidential | public
    status: proposed | reviewed | accepted | archived,
    human_decision: { actor, at, reason } | null,
    redundant_with: list[ref],     # dedupe pointers
    created_at, updated_at
}
```

### 5.3 Generator engine

Generators are **declarative entries in a registry**, not bespoke code.

```python
GeneratorDef {
    id,                            # "problem-statements", "personas", "engagement-plans", ...
    label,
    category,                      # empathize | define | ideate | prototype | test | story | ops | close | critique
    output_schema: type[BaseModel],
    inputs: list[InputSpec],       # which corpus slices + which other Synthesis items
    prompt_template: str,          # version-controlled
    default_count: int,            # ≥3
    range_axis: bool,              # whether to spread along conservative→inverted
    critic_prompt: str | None,
    requires_accepted: list[str],  # upstream items required, e.g. POV requires Persona+Pain
    backed_by_dt_method: str | None  # link to existing dt-methods.ts entry
}
```

Adding a new artifact type = add one registry entry. The UI auto-renders a
candidate-card section under the right category.

### 5.4 Critic

Runs after every Generator. Scores each candidate against:

- evidence quality (count, recency, source diversity)
- internal consistency (vs other accepted items)
- DT discipline (e.g. POV must contain a surprise/insight clause; Persona must
  have a direct customer quote; Engagement Plan with `MVP` fidelity must have a
  validating MVE; etc.)
- novelty vs prior accepted items

Output: per-candidate critique notes inline + a project-level **Gaps Report**
and **Contradictions Report**.

### 5.5 Question Generator (the closed loop)

When Critic flags a gap or contradiction, Question Generator proposes
customer-facing questions to close it. Output bundles include 6Qs draft,
workshop agenda, email draft, standup question, full discovery interview
script. Asked questions are tracked; answers (when they arrive in new
Documents) close the loop and re-trigger affected Generators.

## 7. Engagement Plans

A first-class Synthesis type. Generator picks shape → instantiates template →
tunes durations from corpus signals → produces N variants.

### 7.1 Shape catalog

| Shape | Default duration | Phases | Best for | Default team |
|---|---|---|---|---|
| Brief | 1 wk | Listen → Synthesize → Recommend | Single decision needed | 1 Architect, 0.5 PM |
| Discovery | 2–3 wks | Capture → Orient → Refine → Recommend | "Should we even do this?" | 1 PM, 1 Architect |
| Workshop | 1–2 days | Prep → Run → Synthesize → Follow-up | Stakeholder alignment | 2 facilitators, 1 scribe |
| Hackathon | 2–5 days | Frame → Build → Demo → Retro | Prove a single hard thing fast | 3–6 eng, 1 PM |
| MVE | 2–4 wks | Hypothesis → Build → Measure → Decide | Test one assumption | 2 Eng, 1 PM, 1 Data |
| MVP | 6–12 wks | Discovery → Design → Build → Pilot → Handoff | Production-bound first slice | 4–8 Eng, 1 PM, 1 Design, 1 Architect |
| FDE pod | 8–12 wks rolling | Embed → Iterate → Hand off | Embedded engineering | 2 FDE, 1 PM |
| Pilot → Scale | 12–16 wks | Pilot → Measure → Industrialize | Proven concept expansion | varies |

### 7.2 Plan payload

```python
EngagementPlan {
    shape, duration,
    phases: [{ name, duration, goals, deliverables, exit_criteria, team_shape }],
    milestones: [{ name, anchor_date | relative, evidence_ref? }],
    team: [{ role, fte_pct, stakeholder_slot? }],
    dependencies, assumptions,
    risks_inherited,         # ids from Risk Register
    success_criteria, falsification,
    rationale,               # why this shape, why this duration; cited
    fidelity                 # paper | concept | MVE | MVP
}
```

### 7.3 v1.4 scope

Generator proposes **durations only**, not real dates. Real dates require a
calendar / availability source and arrive when `graph_calendar` data is rich
enough to anchor against (v1.5+).

## 8. Chat (RAG)

- Project-scoped only. Filter on `project_id` at retrieval time.
- Retrieves from **Documents** + **accepted Synthesis** items.
- **Azure AI Search** as the index (Azure-only stack). Embeddings via Azure
  OpenAI `text-embedding-3-large`.
- Every answer carries clickable citation badges to source Documents.
- **Refusal mode:** if the answer cannot be grounded in the corpus, the chat
  states "I don't see this in the project corpus" and offers to search the web
  (separate, clearly-labeled mode) or to draft a Question Generator entry.
- Confidentiality tags propagate into the answer; `confidential` excerpts are
  shown but flagged.

## 9. Office export (v1.5+)

- `python-docx` for `.docx`, `python-pptx` for `.pptx`.
- Templates live in the **customer template** repo, not in the framework
  itself, so each customer can have its own brand.
- The framework ships **default templates** in `backend/app/exporters/templates/`
  so out-of-the-box `npm create core-discovery-app` deployments still produce
  branded artifacts.
- Per artifact mappings: FDE Intake → `.docx`, Solution Pack → `.pptx` (mapped
  cell-by-cell to `solution-pack-template.pptx` sections), Decision Brief → `.docx`,
  Industry Lead Brief → `.docx`, One-Page Brief → `.docx`, Workshop Brief →
  `.docx`.

## 13. Polish & Packaging Standard

Visible in every PR review checklist for v1.3+:

- Empty states have a clear next action.
- Loading states animate; never blank.
- Citations are clickable badges, not URLs in parens.
- Markdown renders with proper typography (headings, code blocks, tables).
- Tables are real tables (sortable where useful), not bullet lists.
- Status colors are semantic and consistent: `proposed`, `accepted`,
  `archived`, `draft-unsupported`.
- No raw markdown viewers in user-facing surfaces; vertex artifacts get
  typed renderers.
- Office exports use the customer template; preview is faithful.

## 15. Glossary

- **Project** — the engagement container, partitioned in Cosmos by `project_id`.
- **Source** — a typed connector that feeds Documents into a project.
- **Document** — the unified ingested-content unit (transcript, email, doc,
  message, …).
- **Synthesis** — the auto-maintained, cited set of artifacts derived from
  Documents.
- **Generator** — a declarative agent that produces N candidate Synthesis
  items.
- **Critic** — the always-on agent that scores candidates and produces Gaps /
  Contradictions reports.
- **The Story** — the human-curated narrative composed from accepted Synthesis
  items; exported as Office artifacts.
- **Discovery** — legacy session model from v1.x; removed in v2.0.
- **DT mode** — the Design Thinking discipline (Empathize / Define / Ideate /
  Prototype / Test) used to organize Synthesis.
- **HVE** — High-Velocity Engineering, the engagement methodology CORE supports.
- **FDE** — Forward-Deployed Engineer; the role triaging pre-engagement work
  under a 7-day SLA.
